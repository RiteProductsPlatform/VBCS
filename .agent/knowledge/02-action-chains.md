# VBCS Skillset — 02: Action Chains (JS vs JSON)

> This is the most critical document. The JS vs JSON chain distinction causes the majority of
> subtle bugs in VBCS applications. Read this before writing any chain.

---

## 1. What Is an Action Chain?

An action chain is the VBCS equivalent of a controller method. It is the **only correct place** for business logic, REST calls, notifications, and variable mutation. Chains are triggered by:
- User events (button click, selection change)
- Page lifecycle events (pageActivated, etc.)
- Other chains calling `Actions.callChain`
- Framework events (`vbNotification`, `application:spShowToast`)

---

## 2. Two Types of Chains

### JS Chains (`.js` files — preferred for complex logic)
AMD modules that extend `ActionChain`. Full JavaScript, async/await, complete control.

```js
define([
  'vb/action/actionChain',
  'vb/action/actions',
  'vb/action/actionUtils',
], (ActionChain, Actions, ActionUtils) => {
  'use strict';

  class myChain extends ActionChain {
    async run(context, params) {
      const { $page } = context;
      // ... logic here
    }
  }

  return myChain;
});
```

### JSON Chains (`.json` files — generated by VBCS visual designer)
Declarative step-by-step action sequences. The VBCS runtime interprets them.

```json
{
  "root": "fireNotificationEventAction1",
  "actions": {
    "fireNotificationEventAction1": {
      "module": "vb/action/builtin/fireNotificationEventAction",
      "parameters": {
        "summary": "Success",
        "message": "Crew saved successfully.",
        "displayMode": "transient",
        "type": "confirmation"
      }
    }
  }
}
```

## 3. CRITICAL: What Parameters Each Type Passes to `vbNotification`

This is the single most common source of bugs in VBCS notification systems.

### JSON `fireNotificationEventAction` → passes ALL parameters

When a JSON chain calls `fireNotificationEventAction`, it passes the full parameter block — including `displayMode`, `type`, `autoTimeout`, and `key` — into the `vbNotification` event payload. The shell page's `showNotificationMessage` chain receives them all.

```json
"parameters": {
  "summary": "Success",
  "message": "Saved.",
  "displayMode": "transient",    ← arrives in event.displayMode
  "type": "confirmation",        ← arrives in event.type
  "autoTimeout": 3000            ← arrives in event.autoTimeout
}
```

### JS `Actions.fireNotificationEvent` → passes ONLY `summary`, `message`, `type`

```js
await Actions.fireNotificationEvent(context, {
  summary: 'Success',
  message: 'Saved.',
  displayMode: 'transient',   ← NOT forwarded to vbNotification event
  type: 'confirmation',       ← forwarded
  severity: 'confirmation',   ← NOT forwarded
  autoTimeout: 3000           ← NOT forwarded
});
```

The `vbNotification` event handler will receive an event object where `event.displayMode`, `event.severity`, and `event.autoTimeout` are all `undefined`, regardless of what you passed to `Actions.fireNotificationEvent`.

**Consequence:** Any condition in `showNotificationMessage.js` that checks `event.displayMode === 'transient'` will **always be false** for JS-based notifications.

**Rule:** In `showNotificationMessage.js`, use `event.displayMode !== 'persist'` (not `=== 'transient'`). This safely treats all notifications as transient unless explicitly marked persistent — which covers JS chains where `displayMode` is undefined.

### For `severity` / `type` in JS chains: always set BOTH

`Actions.fireNotificationEvent` passes `type` to the event but not `severity`. The `showNotificationMessage` handler must read `event.severity || event.type`. Always set both in JS chains for safety:

```js
await Actions.fireNotificationEvent(context, {
  summary: 'Success',
  message: 'Resource saved.',
  severity: 'confirmation',   // for JSON chain compatibility
  type: 'confirmation',       // the one that actually reaches the event
  displayMode: 'transient',
});
```

## 4. Chain Context Object

The `context` parameter passed to `run()` is the most important object in a chain.

```js
async run(context, params) {
  const { $page } = context;       // page-level variables, types, functions
  // $page.variables.xxx            // read/write page variables
  // $page.functions.myHelper()     // call page module functions
}
```

### Context Scope Rules

- In a **shell-page chain**, `context.$page` refers to the **shell page**. Shell page variables are accessed here.
- In a **flow-page chain**, `context.$page` refers to that specific flow page.
- There is no `context.$shell` shortcut from within a flow page chain.

### Context Lifetime — Critical Gotcha

The `context` object is only guaranteed valid during the synchronous execution lifetime of `run()`. After `run()` returns (including after it resolves if async), VBCS may invalidate or recycle the context.

**This means:** Using `context` inside a `setTimeout` callback is unsafe.

```js
// UNSAFE — context may be stale when setTimeout fires
setTimeout(() => {
  Actions.fireDataProviderEvent(context, { ... }); // may fail
}, 4000);
```

**Observed failure mode:** `Actions.fireDataProviderEvent` called with a stale context returns a proxy for the ADP variable, but the proxy's internal data array appears empty — resulting in:
```
[VB ERROR]: Unable to dispatch mutation event — data: [], keys: [3]
event was raised but a valid payload was not provided!
```

The component still displays the items (its view is not updated), but the ADP backing it has been cleared or disconnected. The notification banner never dismisses.

## 5. Calling Other Chains from a Chain

Use `Actions.callChain` to invoke another named chain. Execution waits for the called chain to complete.

```js
// Call another chain on the same page
await Actions.callChain(context, {
  chain: 'loadCrewsChain'
});

// Call a chain with parameters
await Actions.callChain(context, {
  chain: 'filterCrewsChain',
  params: { searchTerm: 'electrical' }
});
```

**Rule:** Chained calls are sequential. If you fire multiple REST calls that don't depend on each other, consider whether they need to be awaited in sequence or can be parallelised with `Promise.all`. VBCS chains don't have built-in parallel execution — you must use native JS `Promise.all` for parallel REST calls.

## 6. Wiring a Chain to a Page Event

In `<page>.json`, the `eventListeners` block maps event names to chains:

```json
"eventListeners": {
  "vbNotification": {
    "chains": [
      {
        "parameters": { "event": "{{ $event }}" },
        "chain": "showNotificationMessage"
      }
    ]
  },
  "messagesBannerSpClose": {
    "chains": [
      {
        "parameters": { "event": "{{ $event }}" },
        "chain": "closeMessageBanner"
      }
    ]
  }
}
```

The `{{ $event }}` expression passes the event object as a parameter named `event` to the chain's `run(context, { event })`.

## 7. Chain Error Handling Pattern

All chains that perform REST calls or mutations must use try/catch/finally. The `finally` block must reset loading/saving state guards.

```js
async run(context) {
  const { $page } = context;
  $page.variables.isSaving = true;

  try {
    const response = await Actions.callRest(context, {
      endpoint: 'CrewsService/createCrew',
      body: payload
    });

    if (response.ok) {
      await Actions.fireNotificationEvent(context, {
        summary: 'Success',
        message: 'Crew created successfully.',
        severity: 'confirmation',
        type: 'confirmation',
        displayMode: 'transient',
      });
    } else {
      await Actions.fireNotificationEvent(context, {
        summary: 'Error',
        message: 'Failed: ' + (response.body?.title || response.statusText),
        severity: 'error',
        type: 'error',
        displayMode: 'transient',
      });
    }
  } catch (error) {
    await Actions.fireNotificationEvent(context, {
      summary: 'Error',
      message: 'Unexpected error: ' + error.message,
      severity: 'error',
      type: 'error',
      displayMode: 'transient',
    });
  } finally {
    $page.variables.isSaving = false;   // always reset, even on error
  }
}
```

**Rules:**
- Always check `response.ok` — a 400/500 response does NOT throw; it resolves with `ok: false`.
- Always put state reset (`isLoading`, `isSaving`) in `finally`, never in try or catch alone.
- Never let an unhandled chain failure leave the UI in a loading/spinner state.

## 8. Payload Handling Best Practices

### Normalise case-insensitive fields from REST responses
Oracle ORDS APIs may return fields in uppercase (`CR_CREWS_ID`) or lowercase (`cr_crews_id`) depending on the version and view definition. Handle both:

```js
const crewId = payload.cr_crews_id || payload.CR_CREWS_ID;
const crewCode = payload.crew_code || payload.CREW_CODE;
```

### Strip nulls before POST, keep nulls for PUT
```js
// For create (POST): omit null/empty fields entirely
for (let key of Object.keys(payload)) {
  if (payload[key] === null || payload[key] === '') delete payload[key];
}

// For update (PUT): send nulls explicitly to clear fields
// (do not strip — ORDS needs to know which fields to clear)
for (let key of Object.keys(payload)) {
  if (payload[key] === '' || payload[key] === undefined) {
    payload[key] = null;
  }
}
```

### Validate before REST call, not after
Always perform field validation (required fields, date order, business rules) before making the REST call. Show an error notification and `return` early. Never rely solely on server-side validation messages — they are often cryptic or missing.

```js
if (startDate && endDate && startDate > endDate) {
  await Actions.fireNotificationEvent(context, {
    summary: 'Validation Error',
    message: 'Start Date cannot be after End Date.',
    severity: 'error',
    type: 'error',
    displayMode: 'transient',
  });
  $page.variables.isSaving = false;
  return;  // early exit — no REST call
}
```
